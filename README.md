# astrbot_plugin_sensitivefilter

群聊敏感词检测插件，支持三种可叠加使用的检测方式 + 群级独立配置。

## 功能特性

**三种检测方式（均可独立开关）**

| 方式 | 说明 |
| --- | --- |
| 本地词库 | 基于 Trie 树的关键词匹配，支持忽略大小写、模糊匹配（识别"敏 感*词"这类拆字绕过写法），以及零宽字符/全角字符归一化 |
| 外部接口 | 默认内置适配 [uapis.cn 的敏感词检测接口](https://uapis.cn/docs/api-reference/post-sensitive-word-quick-check)，开箱即用；也可切换为通用模式适配其他第三方文本审核 API |
| AI 语义检测 | 调用 AstrBot 已配置好的 LLM 提供商，按可自定义的审核 Prompt 对消息做语义级判断，作为关键词匹配的补充 |

三种方式按 **本地词库 → 外部接口 → AI 语义检测** 的顺序依次尝试，命中其中任意一种即视为违规，不会重复触发处理。

**命中后的处理动作（均可独立开关）**

- 撤回消息（目前仅 aiocqhttp 平台实现真正撤回，其他平台会自动跳过撤回但不影响警告/通知）
- 群内警告（@违规者 + 自定义文案）
- 转发通知到指定会话（例如管理员私聊或专门的管理群）

**群级独立配置**

以上"启用本地/接口/AI 检测"和"撤回/警告/通知"这 7 个开关，都支持在全局默认值基础上为单个群单独覆盖，互不影响。每个群还可以维护一份只在本群生效的专属词库。这些群级设置既可以用群内指令管理，也可以直接在 WebUI 插件配置页里以卡片列表的形式逐群查看和编辑——两者操作的是同一份数据。

**访问控制（白名单 / 黑名单）**

跟上面"分群配置"是两套完全独立的机制：分群配置决定"这个群该怎么检测"，访问控制决定"这个群该不该被处理"。支持白名单和黑名单独立开关，可以只开一个，也可以同时开：

- 都关闭（默认）：所有群正常处理
- 只开黑名单：除了黑名单里的群，其余都正常处理
- 只开白名单：只有白名单里的群会被处理，这是纯粹的"仅允许名单"模式
- 都开：白名单优先——只要在白名单里就一定处理，不受黑名单影响；不在白名单里的群一律不处理（不管黑名单怎么设）

## 代码结构

```
astrbot_plugin_sensitivefilter/
  metadata.yaml        插件元数据
  _conf_schema.json     插件配置 Schema（WebUI 可视化配置，含分群配置卡片）
  main.py               插件入口：事件监听、管理指令、命中后的处理动作
  word_matcher.py        本地词库匹配（Trie 树）
  api_checkers.py        外部接口检测（uapis.cn 适配 + 通用模式）
  llm_checker.py         AI 语义检测（调用 LLM Provider）
  utils.py               跨模块共享的小工具函数
  tests/                 开发期回归测试，不影响插件运行，详见 tests/README.md
```

`main.py` 只负责编排（决定先用哪种方式检测、命中后做什么），具体的检测逻辑都下沉到 `api_checkers.py` / `llm_checker.py` / `word_matcher.py` 三个独立模块里，互不依赖，方便单独测试和替换。群级覆盖配置不再使用独立的 JSON 文件存储，而是直接落在插件配置的 `group_overrides` 字段里（详见下方"分群配置"）。

## 安装

将本插件文件夹放入 `AstrBot/data/plugins/astrbot_plugin_sensitivefilter`，在 WebUI 插件管理页重载即可。

## 重要提示：关于词库内容

出于安全考虑，本插件**不会预置任何真实的违禁/敏感词内容**，默认词库仅有一个示例词 `测试敏感词`，用于验证插件是否正常工作。请管理员根据自己社群的实际需求，通过下方指令或 WebUI 配置页自行填写需要拦截的词语。

## 管理指令

以下指令默认仅管理员可用：

```
/敏感词 帮助                     查看指令列表
/敏感词 添加 <词>                 全局词库新增一个词
/敏感词 删除 <词>                 全局词库删除一个词
/敏感词 列表                     查看全局词库（最多显示前 50 个）
/敏感词 本群添加 <词>             仅本群额外生效的专属词
/敏感词 本群删除 <词>             删除本群专属词
/敏感词 设置 <项> <on/off/默认>   为本群单独覆盖某个开关
/敏感词 状态                     查看本群当前每一项的生效值及是否被覆盖
/敏感词 白名单 开启|关闭|添加本群|删除本群|列表
/敏感词 黑名单 开启|关闭|添加本群|删除本群|列表
```

`/敏感词 设置` 支持的 `<项>`：`总开关`、`本地`、`接口`、`ai`、`撤回`、`警告`、`通知`。
第二个参数填 `on`/`off` 表示开启或关闭，填 `默认` 表示取消本群覆盖、跟随全局配置。

这些指令操作的数据和下方"分群配置"/"访问控制" WebUI 卡片是**同一份数据**，两种方式可以混用：比如先在群里用指令快速开关一个选项，之后再去 WebUI 里统一查看/批量调整所有群的配置。

## 访问控制（WebUI / 指令）

插件配置页最上方的"访问控制"区块（对应 `whitelist_*`/`blacklist_*` 配置项），决定"这个会话该不该被处理"，跟下面的"分群配置"（决定"该怎么处理"）是两套完全独立的机制：

- **白名单**：开启后，只有白名单 umo 列表里的会话会被检测，其余一律不响应
- **黑名单**：开启后，黑名单 umo 列表里的会话会被完全跳过
- 两者都开启时，**白名单优先**——命中白名单就一定处理，不再看黑名单

群里发送 `/敏感词 白名单 添加本群` / `/敏感词 黑名单 添加本群` 可以快速把当前群加入对应名单。注意"添加本群"只操作列表内容本身，不会自动打开名单的总开关，需要额外用 `/敏感词 白名单 开启`（或 WebUI 里的开关）才会真正生效；也可以直接在 WebUI 里维护 umo 列表。

## 分群配置（WebUI）

插件配置页有一个"分群配置"区块（对应 `_conf_schema.json` 里的 `group_overrides`），以卡片列表的形式管理每个群的独立设置，点击"+ 添加条目"即可新增一个群：

- **umo（unified_msg_origin）**：填写该会话的 `unified_msg_origin`，格式如 `aiocqhttp:GroupMessage:123456`；在群里发送 `/敏感词 状态` 可以直接看到当前群的 umo，复制粘贴进来即可
- **插件总开关 / 本地词库检测 / 外部接口检测 / AI 语义检测 / 命中后撤回 / 命中后警告 / 命中后通知**：每项都是「跟随全局 / 开启 / 关闭」三选一，默认"跟随全局"表示该群没有单独设置，使用上方全局默认值
- **本群专属敏感词**：只在这个群生效的额外词库，会与全局词库叠加检测

未在这个列表里出现的群，所有设置都按全局默认值执行。如果一个群的所有选项都改回"跟随全局"并且专属词库也清空了，下次通过指令修改时插件会自动把这条多余的记录清理掉，避免列表越攒越长。

> 注意：分群配置里没有"是否处理这个群"的选项——那是上面"访问控制"的职责。两者故意分开，避免"删除分群配置的自定义设置"和"把这个群踢出白名单"这两件不相关的事互相影响。

## 配置说明（WebUI 插件配置页）

除了"分群配置"（template_list，单独说明见上），全局默认配置通过 `_conf_schema.json` 在 WebUI 中可视化编辑，分成五个分组卡片：

1. **访问控制**：白名单/黑名单开关及对应的 umo 列表
2. **基础设置**：插件总开关、本地词库检测开关、全局词库、大小写/模糊匹配、命中后是否阻止事件继续传播
3. **处理动作默认值**：撤回/警告/警告文案/通知开关/通知接收会话列表
4. **外部接口检测**：接口类型、地址、密钥、请求方式、请求头、字段路径、超时时间（滑块）
5. **AI 语义检测**：开关、审核使用的模型提供商（点击按钮从已配置的 Provider 中选择，无需手填 ID）、审核 Prompt 模板

警告文案支持 `{sender}`、`{word}`、`{source}` 三个占位符；AI 审核 Prompt 模板必须包含 `{text}` 占位符（用于插入待审核的消息原文）。

> 这五个分组对应 `_conf_schema.json` 里的五个嵌套 object（`access_control`/`basic`/`actions`/`api_detection`/`llm_detection`），插件代码内部通过统一的 `_cfg(key)` / `_set_cfg(key, value)` 方法读写，不需要在业务逻辑里关心具体的分组路径。

## 关于外部接口检测

插件内置了两种外部接口模式（由配置项 `api_provider` 切换）：

### 1. uapis_profanitycheck（默认，推荐）

专门适配 [uapis.cn 的「敏感词检测（快速）」接口](https://uapis.cn/docs/api-reference/post-sensitive-word-quick-check)，开箱即用：

- `api_url` 默认已经填好为 `https://uapis.cn/api/v1/text/profanitycheck`，一般不需要改
- `api_key` 选填：不填则使用该平台的免积分访客额度调用；填写后会通过 `Authorization: Bearer <api_key>` 携带，额度更高更稳定。可以在 [uapis.cn 控制台](https://uapis.cn/console/api-keys) 创建密钥
- 命中判断逻辑：接口返回 `status == "forbidden"` 时视为命中，命中的具体词语来自响应里的 `forbidden_words` 字段，会显示在群内警告文案的"来源"里
- 该接口支持简繁体中文混合检测，词库由 uapis.cn 维护更新，作为本地词库的补充比较合适（本地词库命中速度更快、可自定义；这个接口词库更全、免维护）

只需要把 `api_enabled` 打开、确认 `api_provider` 为 `uapis_profanitycheck`，即可立即生效，不需要额外配置字段路径。

### 2. generic（通用模式）

如果想接入其他任意第三方审核接口，把 `api_provider` 改成 `generic`，再配置：

插件会把消息文本放进请求体的一个字段中（字段名可配置，默认 `text`）发给配置的 `api_url`，然后从返回的 JSON 中按"点路径"取出一个布尔字段判断是否命中（默认路径 `hit`），以及一个可选的文本字段作为命中原因（默认路径 `reason`）。

例如某接口返回 `{"result": {"is_violation": true}, "msg": "广告"}`，则可以把 `api_hit_path` 配置为 `result.is_violation`，`api_reason_path` 配置为 `msg`。

## 已知限制

- 自动撤回目前只在 aiocqhttp（如 Napcat/Lagrange 等协议端）上实现，其他平台暂不支持撤回，命中后只会发送警告/通知。
- 不支持"多次违规自动禁言"等升级处罚机制，目前仅做撤回 + 警告 + 通知三个动作。如有需要可以在 `_handle_violation` 里自行扩展。
- 本地词库匹配是简单的逐位置扫描实现，复杂度约为 O(消息长度 × 平均词长)，足以应对聊天消息场景，但不适合用来扫描超大文本。
- "分群配置"用的是 AstrBot 插件配置 Schema 里的 `template_list` 类型；如果直接编辑插件配置 JSON 文件而不是通过 WebUI/指令，新增的条目需要带上 `"__template_key": "group_override"` 字段，否则 WebUI 打开时会提示找不到对应模板（通过指令或 WebUI 正常操作不会遇到这个问题，这里只是给需要手动改文件的人提个醒）。
